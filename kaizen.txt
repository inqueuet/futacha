================================================================================
futacha コード分析レポート — バグ・フリーズ要因の指摘と改善提案
分析日: 2026-06-11
================================================================================

総評:
本コードベースは防御的修正(FIX: コメント)が多数入っており、典型的なフリーズ原因
(busy-wait、無制限読み込み、CancellationException の握りつぶし、デッドロック)は
ほぼ対策済み。while(true) ループは全箇所サスペンドポイントを含み busy-wait なし。
以下は残存する問題を重要度順に列挙したもの。

--------------------------------------------------------------------------------
【高】1. スレ表示の LazyColumn key に重複排除なしの post.id を使用(クラッシュ経路)
--------------------------------------------------------------------------------
場所:
  shared/src/commonMain/kotlin/ui/board/ThreadContentLayout.kt:147-149
    itemsIndexed(items = page.posts, key = { _, post -> post.id })
  shared/src/commonMain/kotlin/parser/ThreadHtmlParserCore.kt:352-354, 380-381

問題:
  パーサーは post.id を HTML の No.XXXX / data-res からそのまま取り出しており、
  重複排除がどこにもない。LazyColumn は同一 key が2つあると
  IllegalArgumentException: "Key was already used" を投げて画面ごとクラッシュする。
  カタログは distinctBy { item.id } で防御済み(CatalogViewTransformer.kt:56)、
  ツリー表示も visited セットで防御済み(ThreadTreeContent.kt:69-73)なのに、
  通常のスレ表示だけ無防備。アーカイブサイト由来のHTML(ArchiveSearch)も同じ
  パーサーで表示するため、重複 No. に遭遇する確率は無視できない。

対策:
  パース後に distinctBy { it.id } する、または key を "$index-${post.id}" 形式にする。
  修正コストは極小。最優先で対応推奨。

--------------------------------------------------------------------------------
【中】2. 端末AIリクエスト: タイムアウトが Mutex の内側にあり、ロック待ちは無制限
--------------------------------------------------------------------------------
場所:
  shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:483-484
    requestMutex.withLock { withTimeoutOrNull(timeoutMillis) { ... } }

問題:
  withTimeoutOrNull がロック取得後に始まるため、ロック待ち自体には何のタイム
  アウトも効かない。要約リクエストは1件最大180秒、可用性チェックは90秒ロックを
  保持するので、要約中に設定画面を開くと可用性表示が数分間「確認中」のまま、
  後続の要約・荒らし判定もすべて待たされる。呼び出し側がキャンセルすれば解放
  されるため永久フリーズではないが、「AIまわりが固まって見える」最有力原因。

対策:
  withTimeoutOrNull を withLock の外側に出し、ロック待ちを含めて計時する。

--------------------------------------------------------------------------------
【中】3. AIコマンドが1本の collect で直列実行され、履歴更新中は最大60秒無反応
--------------------------------------------------------------------------------
場所:
  shared/src/commonMain/kotlin/ui/FutachaApp.kt:365-369
  shared/src/commonMain/kotlin/ui/FutachaAiCommandRouter.kt:90-104
    (AI_HISTORY_REFRESH_TIMEOUT_MILLIS = 60_000L)

問題:
  FutachaAiCommandBridge.commands のコレクタは1本で、RefreshHistory コマンドは
  collect 内で refresher.refresh(...) を最大60秒同期的に待つ。この間、
  Siri / Watch / ディープリンク由来の後続コマンドはチャネル(容量128)に溜まる
  だけで無反応に見え、あふれた分は黙って破棄される。

対策:
  長時間かかるアクションだけ別の Job に逃がして collect ループを即時返す。

--------------------------------------------------------------------------------
【中】4. WebM サムネイル生成がメインスレッドの WKWebView で直列実行(iOS)
--------------------------------------------------------------------------------
場所:
  shared/src/iosMain/kotlin/ui/image/IosVideoFrameFetcher.kt:92, 217-254, 362-395

問題:
  Semaphore(permits = 1) で全 WebM サムネイルが直列化され、1件あたり最大10秒
  (HTMLロード + 100ms×25回のJSポーリング + スナップショット)。処理はすべて
  メインキュー上で動く。WebM が多いカタログではスクロール中のジャンク(コマ落ち)
  とサムネイルの数十秒単位の遅延が発生し、「固まった」ように見える。

対策:
  ポーリング間隔/回数の見直し、AVFoundation で扱える形式への変換検討、
  生成結果のディスクキャッシュ化など。

--------------------------------------------------------------------------------
【中】5. 正規表現の「ReDoSタイムアウト」が実際には機能しない(偽の安全網)
--------------------------------------------------------------------------------
場所:
  shared/src/commonMain/kotlin/parser/ThreadHtmlParserCore.kt:306-333 (extractBetween)

問題:
  Regex.find は CPU バウンドでサスペンドポイントを含まないため、
  withTimeoutOrNull(500) はマッチング実行中に割り込めない(終わってから結果を
  捨てるだけ)。現在は各パターンの長さ制限([^>]{0,500} 等)が実質の防御なので
  即座の危険はないが、コメントの「タイムアウト」は誤った安心感を与える。
  マッチは AppDispatchers.parsing = Dispatchers.Default.limitedParallelism(2)
  (AppDispatchers.kt:52)で走るため、バックトラッキングするパターンが将来
  1つ追加されただけで、2本詰まれば全画面のパース処理(カタログ・スレ・履歴
  JSONデコード)が停止し、UIは生きているのに何も読み込まれない実質フリーズになる。

対策:
  入力長の事前制限を防御の本線として維持し、パターン変更時はバックトラッキング
  検査を行う。誤解を招くコメントは修正する。

【低中】6. Wear連携で Tasks.await が IO スレッドをブロック
--------------------------------------------------------------------------------
場所:
  app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:514-520

問題:
  Tasks.await(task, 10s) を Dispatchers.IO 上で実行。スナップショット1回の送信で
  connectedNodes + putDataItem + ノード数分の sendMessage を逐次ブロッキング待ち
  (最悪10秒×N)。リトライが重なると Dispatchers.IO のスレッドを長時間占有する。

対策:
  kotlinx-coroutines-play-services の非ブロッキング Task.await() へ置き換え。

--------------------------------------------------------------------------------
【低中】7. 読み上げランナーが外側スコープのキャンセルを握りつぶす形
--------------------------------------------------------------------------------
場所:
  shared/src/commonMain/kotlin/ui/board/ThreadReadAloudRunnerSupport.kt:82-95

問題:
  TimeoutCancellationException を CancellationException より先に catch して通常
  リターンするため、外側の withTimeout 起因のキャンセルを飲み込む形になっている。
  また wasCancelledByUser() が真のときに CancellationException を握りつぶして
  正常リターンするため、ユーザー停止と親スコープのキャンセルが同時の場合に
  キャンセル済みスコープで後続コードが走る(次のサスペンドで再スローされるため
  実害は限定的だが、読み上げ状態の不整合につながり得る)。

対策:
  自分のタイムアウトかを判別する、または return 前に
  currentCoroutineContext().ensureActive() を挟む。

--------------------------------------------------------------------------------
【低】8. iOS の IO ディスパッチャが固定4スレッド
--------------------------------------------------------------------------------
場所:
  shared/src/iosMain/kotlin/util/AppDispatchers.ios.kt:8-9
    newFixedThreadPoolContext(4, "FutachaIo")

問題:
  iOS の FileSystem は全ファイルI/O(NSData 全量読み等のブロッキング呼び出し)を
  このプールで実行する。バックグラウンド更新(maxConcurrency=2)+ スレ自動保存
  (最大300メディア・512KBバッファ)+ Watch スナップショット構築が重なると
  4スレッドが埋まり、全体が詰まりやすい。

対策:
  容量増、または用途別の limitedParallelism 分割。

--------------------------------------------------------------------------------
【低】9. モデレーション IPC のペイロード上限が Binder 限界に近い
--------------------------------------------------------------------------------
場所:
  shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:95
    (AI_IPC_MODERATION_MAX_TOTAL_MESSAGE_CHARS = 120_000)

問題:
  1チャンク最大 120,000 文字は Parcel 上 UTF-16 で約240KB。Binder のトランザク
  ションバッファ(約1MB、プロセス内共有)に対して余裕が薄く、他の Binder
  トラフィックと重なると TransactionTooLargeException の余地がある。失敗時は
  RemoteException → finish(null) で静かに機能停止に見える。

対策:
  チャンク上限を 64KB 相当などに引き下げる。

--------------------------------------------------------------------------------
【低】10. iOS ストレージ: aiCommandState だけコンストラクタで同期読み
--------------------------------------------------------------------------------
場所:
  shared/src/iosMain/kotlin/state/AppStateStore.ios.kt:77

問題:
  他のキーはすべて遅延ロードなのに AI_COMMAND_KEY だけ初期化時に NSUserDefaults
  を同期読みし、その後 deferred load でも再度読み直している。実害はほぼないが、
  非対称な初期化パスは将来の競合バグの温床。

対策:
  deferred load 側に統一する。

================================================================================
問題なしと確認した領域(参考)
================================================================================
- ネットワーク読み込み: ゼロリード時のバックオフ+回数上限・総量上限・総時間
  タイムアウト完備(HttpBoardApiResponseSupport / ArchiveSearch / VersionChecker)
- HTMLパーサー: 全正規表現に長さ上限済み(ReDoS対策。ただし上記5の注意あり)
- CancellationException の再スロー: 全域で徹底
- Mutex 構成: 履歴/ボード/スクロールで分離されデッドロック経路なし
- PersistentCookieStorage: トランザクション差分適用・ネスト対応まで堅牢
- BackgroundRefreshManager(iOS BGTask): メインスレッド集約 + expirationHandler 対応
- TextSpeaker(Android TTS): QUEUE_FLUSH 中断時の onStop での continuation 解放、
  初期化コールバック未達端末向けタイムアウトまでケア済み
- WatchSnapshotStore / PhoneCommandClient(Wear): 世代比較・CASガード・メイン直列化
- ThreadSaveService: サイズ・時間・並列数・ゼロリードすべて上限付き、
  キャンセル/失敗時クリーンアップ有り
- WCSession デリゲート: DispatchQueue.main.sync を意図的に回避済み
- HistoryRefresher: 二重実行防止 + アボート時フラッシュ(NonCancellable)
- Compose ポインタ処理ループ: すべてサスペンドし busy-wait なし
- FutachaAppleIntelligenceBridge(Swift): タスクキャンセル管理・45秒タイムアウト有り
