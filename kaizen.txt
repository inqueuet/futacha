# Futacha 改善メモ: フリーズ・負荷・遅延リスク

## 高: 履歴を 1 本の JSON として全件読み書きしている

- 対象:
  - shared/src/commonMain/kotlin/state/AppStateStore.kt:150
  - shared/src/commonMain/kotlin/state/AppStateHistoryMutationSupport.kt:161
  - shared/src/commonMain/kotlin/state/AppStateHistoryCoordinatorSupport.kt:142
  - shared/src/androidMain/kotlin/state/AppStateStore.android.kt:364
- 内容:
  - 履歴 Flow は `historyJson` 全体を decode する。
  - スクロール位置更新でも履歴全体を `map` し、全件 JSON に encode して保存する。
  - Android は DataStore の文字列 preference、iOS は NSUserDefaults 相当の大きな文字列更新になる。
- リスク:
  - 履歴件数が増えるほど、スレ閲覧中のスクロール保存や履歴更新後の反映が重くなる。
  - DataStore/NSUserDefaults の更新頻度が高いと、端末によっては UI が固まったように見える。
- 改善案:
  - 履歴を単一 JSON ではなく、スレ単位またはページ単位で保存する。
  - スクロール位置は別ストアに分離し、最後に開いた数十件だけを永続化する。
  - 少なくともスクロール位置更新はメモリ上に集約し、アプリバックグラウンド化やスレ離脱時にまとめて保存する。

## 高: 自動保存がメディア込みで重い

- 対象:
  - shared/src/commonMain/kotlin/ui/board/ThreadAutoSaveRunnerSupport.kt:10
  - shared/src/commonMain/kotlin/ui/board/ThreadAutoSaveRunnerSupport.kt:73
  - shared/src/commonMain/kotlin/ui/board/ThreadScreenSaveExecutionBindingsSupport.kt:396
  - shared/src/commonMain/kotlin/service/HistoryRefreshAutoSaveSupport.kt:77
- 内容:
  - スレ画面の自動保存は最大 300 メディア、90 秒上限で走る。
  - 履歴更新中の auto-save も `ThreadSaveService.saveThread()` を使い、本文とメディア保存を行う。
  - 指紋が変わったスレでは、通常閲覧中にも 60 秒間隔で保存候補になる。
- リスク:
  - 画像/動画が多いスレで通信、ディスク IO、CPU、バッテリー消費が大きい。
  - 手動保存、履歴更新、バックグラウンド更新と重なると固まりやすい。
- 改善案:
  - 全メディア保存は維持し、自動保存だけスレ単位の安定フォルダへ差分更新し、保存済みメディアの再ダウンロード/再書き込みを避ける。
  - 第一段階では `autosaved_threads/<board-thread>/...` を使い、auto-save のたびに世代フォルダを作り直さない。前回 metadata の `original...Url -> local...Path` を読んで、実ファイルが残っているメディアは再利用する。
  - 手動保存は今まで通り世代フォルダ内にメディアを置き、ユーザーが見る保存フォルダと履歴エクスポートの自己完結性を保つ。
  - 差分更新では保存開始時に既存フォルダを削除しない。`prepareThreadSaveStorageTarget()` は auto-save 差分モードだけ「create only」に分ける。
  - スレ画面 auto-save の `writeInitialMetadataBeforeMedia` は安定フォルダと相性が悪い。前回正常 `metadata.json` を途中状態で上書きしないよう、差分モードでは初期 metadata を書かない。
  - 差分保存前には `metadata.json.backup` を残し、`metadata.json` が壊れていた場合は `loadThreadMetadata()` が backup へフォールバックする。
  - `SavedThreadMetadata.toThreadPage()`、履歴アーカイブ export/import、削除処理は既存の storageId 相対パス前提を維持できる。`cache://` 導入は第二段階に回す。
  - スレ画面 auto-save と履歴更新 auto-save が同じスレを同時に扱う可能性があるため、安定 storageId 単位で `ThreadStorageLockRegistry` を使い、同一フォルダ差分更新を直列化する。
  - 前回 metadata に存在し、新しい metadata から参照されなくなったメディアだけ削除すれば、安定フォルダでも不要ファイルの蓄積を抑えられる。
  - `maxMediaItems` を下げる案は、全メディア保存要件と矛盾するので採用しない。Wi-Fi/充電条件は追加設定としては有効だが、根本解決ではない。

- 10回調査メモ:
  1. 現状は毎回 `buildThreadSaveGenerationStorageId()` で新しい世代フォルダを作るため、同じメディアでも保存世代ごとに再取得・再書き込みになる。
  2. スレ画面 auto-save は 60 秒間隔、最大 300 メディア、90 秒上限、並列 1。履歴更新/WorkManager/Watch/iOS BG も `ThreadSaveService.saveThread()` を通る。
  3. UI の手動履歴更新は auto-save 予算 0 / 件数 0 なので、主な負荷源はスレ画面 auto-save とバックグラウンド更新。
  4. `ThreadSaveService` のインスタンス内 `mediaWriteLocks` は別インスタンス間を守れない。安定フォルダを同じスレの auto-save で共有するなら、storageId 単位のプロセス共通ロックが必要。
  5. `FileSystem` は `../` を拒否するため、世代フォルダから `../media_cache` を参照する設計は使えない。
  6. 既存の `toThreadPage()` は storageId 相対パス前提。安定フォルダ差分更新ならこの前提を維持できるが、`cache://` 方式に進むなら明示的な解決処理が必要。
  7. `SavedThreadRepository.addThreadToIndex()` は同一スレの古い世代フォルダを削除する。新しい auto-save の storageId を安定IDにすれば、初回成功時に旧世代は掃除され、以後は同じフォルダを更新できる。
  8. `deleteThread()` / `deleteAllThreads()` は storageId を消す。安定フォルダ差分更新なら削除処理を大きく変えずに済む。
  9. 履歴アーカイブは metadata のローカルパスを `$storageId/$relativePath` としてコピーする。安定フォルダ差分更新ならこの前提を維持できるが、`cache://` を導入すると export/import も同時対応が必要になる。
  10. グローバルURLキャッシュは重複削減が最大だが参照管理が複雑。まず安定フォルダ差分更新にすれば、影響範囲を抑えたまま同一スレの再保存コストを大きく減らせる。

## 高: 保存インデックス更新がロック内で重い

- 対象:
  - shared/src/commonMain/kotlin/repository/SavedThreadRepository.kt:84
  - shared/src/commonMain/kotlin/repository/SavedThreadRepository.kt:360
  - shared/src/commonMain/kotlin/repository/SavedThreadRepositoryIndexSupport.kt:150
- 内容:
  - `addThreadToIndex()` は index lock を取り、index 全体を読み書きする。
  - 保存時に `index.json.tmp`、`index.json.backup`、`index.json` の 3 ファイルを書き込む。
  - ロック待ち、ロック保持ともに最大 30 秒のタイムアウトがある。
- リスク:
  - 自動保存や手動保存が連続すると、保存インデックスで詰まりやすい。
  - ロック中にファイル IO が遅い端末では、保存完了や履歴更新の反映が遅くなる。
- 改善案:
  - インデックスを全件 JSON ではなく、スレ単位の manifest に分割する。
  - backup 書き込みは頻度を下げる、または atomic rename が使える環境では temp -> rename に寄せる。
  - auto-save の初期 metadata 登録と完了 metadata 登録をまとめ、index 更新回数を減らす。

## 中: 履歴更新がプロセス全体で直列化されている

- 対象:
  - shared/src/commonMain/kotlin/service/HistoryRefresher.kt:81
  - shared/src/commonMain/kotlin/service/HistoryRefresher.kt:117
  - app-android/src/main/java/com/valoser/futacha/FutachaApplication.kt:107
- 内容:
  - `HistoryRefresher` は static な `processRefreshMutex` で、UI、WorkManager、Watch 由来の更新を全体で 1 本に制限している。
  - Android のアプリ singleton は `maxConcurrency = 1` で作られている。
- リスク:
  - バックグラウンド更新中にユーザーが手動更新すると `RefreshAlreadyRunningException` になりやすい。
  - 大きな履歴を持つユーザーほど、手動更新が「反応しない」「遅い」と感じやすい。
- 改善案:
  - foreground の手動更新を優先し、background/watch 更新はキャンセルまたは延期する。
  - UI には「バックグラウンド更新中」状態を表示し、失敗ではなく待機/再試行にする。
  - Android singleton の `maxConcurrency = 1` を、端末性能やネットワーク状態に応じて 2 程度にする。

## 中: Wear 同期が常時起動し、履歴変更ごとに全履歴を走査する

- 対象:
  - app-android/src/main/java/com/valoser/futacha/FutachaApplication.kt:116
  - app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:101
  - app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:521
  - app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:721
- 内容:
  - `WatchSyncManager.start()` がアプリ起動時に常時呼ばれる。
  - 履歴更新ごとに fingerprint 用のリストを全件生成し、snapshot JSON を作って Wear Data Layer 送信を試みる。
  - preview 用 metadata も最大 8 件ロードする。
- リスク:
  - Wear を使っていないユーザーにも CPU/IO/送信試行の負荷が乗る。
  - 履歴件数が増えるほど、履歴変更時の裏処理が重くなる。
- 改善案:
  - Wear 同期を設定で opt-in にする。
  - ペアリング済み Wear 端末が確認できる場合だけ collector を開始する。
  - fingerprint を全履歴ではなく、Watch に送る上限件数に絞って作る。

## 中: Watch snapshot の retry に上限がない

- 対象:
  - app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:124
- 内容:
  - Watch snapshot stream が失敗した場合、`retryWhen` で delay 後に再試行し続ける。
  - retry 回数の上限や、一定回数失敗後に Watch 同期を一時停止する仕組みがない。
- リスク:
  - Wear / Google Play services / Data Layer の状態が悪い端末で、ログ出力と再試行が長時間残る。
  - Watch を使っていないユーザーでも、失敗状態によってはバックグラウンド負荷が続く。
- 改善案:
  - retry 回数または総 retry 時間に上限を設ける。
  - 連続失敗時は Watch 同期を一時停止し、次回アプリ起動、設定変更、端末接続検出時に再開する。
  - 失敗理由が Wear 非対応や Play services 不在の場合は collector 自体を止める。

## 中: カタログ画像の拡張子フォールバックが通信を増幅しやすい

- 対象:
  - shared/src/commonMain/kotlin/ui/board/CatalogPreviewImageComponents.kt:55
  - shared/src/commonMain/kotlin/ui/image/ImageLoaderProvider.kt:140
  - shared/src/commonMain/kotlin/ui/image/ImageLoaderProvider.kt:202
- 内容:
  - 画像ロード失敗時、最大 5 回の拡張子フォールバックを試す。
  - 動画候補も最大 2 回、タイムアウト付きで試す。
  - negative cache はあるが、初回表示や高速スクロールでは新規 URL ごとに試行される。
- リスク:
  - 欠落 URL が多い板で通信回数が増える。
  - カタログ高速スクロール時にネットワークとバッテリー負荷が増える。
- 改善案:
  - カタロググリッドでは `maxAttempts` を 1-2 に下げる。
  - グリッド表示では動画フォールバックを無効化し、スレ画面やプレビューだけで試す。
  - サーバー/板ごとの失敗率を見て、失敗が多い場合は一定時間フォールバックを停止する。

## 中: 共有 HttpClient を閉じてしまう事故が起きやすい API 形状

- 対象:
  - shared/src/commonMain/kotlin/repo/BoardRepositoryFactory.kt:8
  - shared/src/commonMain/kotlin/repo/BoardRepository.kt:405
  - shared/src/commonMain/kotlin/network/HttpBoardApi.kt:525
- 内容:
  - `createRemoteBoardRepository(httpClient)` で外部注入した client でも、Repository を close すると `HttpBoardApi.close()` が client 本体を close する。
  - 現行 UI 経路では shared Repository を閉じないガードがあるが、別経路で close されると共有 client が巻き添えになる。
- リスク:
  - UI、履歴更新、保存、バックグラウンド処理が同じ client を使っている場合、どこかの close で全体の通信が壊れる。
  - 再現条件が複雑なため、ランダムな通信失敗やフリーズに見えやすい。
- 改善案:
  - `HttpBoardApi` に `ownsClient` を持たせ、外部注入 client は close しない。
  - `createRemoteBoardRepository()` と `createRemoteBoardRepository(httpClient)` を内部的に所有権付きで分ける。
  - Repository close の単体テストで、共有 client が閉じられないことを保証する。

## 低: 読み上げ・検索・派生データ生成は全投稿を走査する場面がある

- 対象:
  - shared/src/commonMain/kotlin/ui/board/ThreadScreenDerivedRuntimeSupport.kt:130
  - shared/src/commonMain/kotlin/ui/board/ThreadScreenDerivedRuntimeSupport.kt:159
  - shared/src/commonMain/kotlin/ui/board/ThreadReadAloudSupport.kt:30
- 内容:
  - 検索開始、読み上げ開始、投稿更新時に全投稿を走査して派生データを作る。
  - 実処理は parsing dispatcher に逃がされているため直撃度は低い。
- リスク:
  - 2000 投稿級のスレで検索/読み上げを頻繁に切り替えると、CPU 負荷が出る。
- 改善案:
  - 投稿本文の plain text 化結果をスレ単位で cache し、検索/読み上げで共有する。
  - 検索結果や読み上げセグメントは page revision 単位で再利用する。

## 低: Android AI 処理が完全直列で待ちを作りやすい

- 対象:
  - shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:490
  - shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:497
  - shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:941
  - shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:983
- 内容:
  - Android AI remote session は `requestMutex.withLock` でリクエストを 1 本ずつ処理する。
  - スレ要約や投稿分類の inference 中は、他の AI リクエストが待たされる。
  - Service scope 自体は `onDestroy` で cancel されるためリークの疑いは低い。
- リスク:
  - 大きいスレの要約中に、荒らし判定や可用性確認など別 AI 操作が遅く見える。
  - AI 処理のタイムアウト待ちが長い場合、UI 側では「AI が固まった」ように見える可能性がある。
- 改善案:
  - 可用性確認、要約、投稿分類で queue を分ける。
  - 画面側に「AI 処理待機中」の状態を明示する。
  - 重い処理はキャンセルボタンや最新リクエスト優先の置き換えを強める。

## 優先対応順

1. 履歴の全件 JSON 保存をやめる、またはスクロール位置保存だけ分離する。
2. 自動保存のメディア保存をスレ単位の安定フォルダ差分更新にし、保存済みメディアの再ダウンロード/再書き込みを避ける。
3. SavedThreadRepository の index 更新回数とロック内 IO を減らす。
4. Foreground 手動更新と background/watch 更新の優先順位を分ける。
5. Wear 同期を opt-in または端末検出後に限定する。
6. Watch snapshot retry に上限と一時停止条件を入れる。
7. カタログ画像フォールバックの試行回数を下げる。
8. 共有 HttpClient の所有権を API と型で明示する。
9. AI 処理の queue と待機表示を整理する。
