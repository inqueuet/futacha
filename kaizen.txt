■ 最優先(フリーズの直接原因)
--------------------------------------------------------------------------------

[P1-1] iOS: WebM サムネイル生成が WKWebView をメインスレッドで実行
  場所: shared/src/iosMain/kotlin/ui/image/IosVideoFrameFetcher.kt:216-252
  内容:
    - webm 添付のサムネイルを、メインキュー上で WKWebView 生成 → HTML ロード
      → WebView 内で動画ダウンロード/デコード → JS ポーリング(100ms x 25回)
      → スナップショット撮影、という手順で生成している。
    - WKWebView の生成・ロードだけで数十〜数百ms メインスレッドを占有するため、
      webm 付きレスが画面に入るたびに画面全体がフリーズする。
      iOS でのフリーズの最有力原因。
  追加の問題:
    - この処理には全体タイムアウトがない。didFinishNavigation も didFail も
      呼ばれずロードが止まると continuation が永遠に resume されず、
      webKitThumbnailSemaphore(permits=1、:92)を握ったままになる。
      → 以後セッション中すべての webm サムネ取得が永久に詰まる。
      (AVFoundation 側 :134 には 20 秒タイムアウトがあるが WebKit 側にはない)
  修正方針:
    - withTimeoutOrNull で全体タイムアウトを付け、セマフォ解放を保証する(簡単・効果大)。
    - 可能なら WKWebView 方式自体をオフメイン化または事前生成/再利用に変更。

[P1-2] AI 投稿フィルタ有効時: レス1件ごとに全画面更新+サービス bind/unbind
  場所: shared/src/commonMain/kotlin/ui/board/ThreadScreenContentHost.kt:27
        (THREAD_AI_POST_MODERATION_BATCH_SIZE = 1)
        shared/src/androidMain/kotlin/ai/OnDeviceAiService.android.kt:447-545
  内容:
    - バッチサイズ1のため、レス1件分類するたびに AiPostModerationUiState
      (新しい results リスト入り)が発行され(:358-365)、ThreadContent 全体が
      再コンポーズ → aiHiddenPostState の O(n) 再計算(:380-393)も毎回再実行。
      1000 レスのスレでは数百回繰り返される。
    - Android では requestRemoteAi が 1 リクエストごとに bindService/unbindService
      するため、レス1件 = 1回の IPC 接続確立+切断。数百回の bind/unbind が走る。
    - 処理中ずっと CPU を食い続け、端末によっては固まったように見える。
  修正方針:
    - バッチサイズを増やす。UI への結果反映を間引く(例: 10件ごと/完了時のみ)。
    - サービス接続を一定時間維持して使い回す。

■ 高優先(スクロール時のカクつき: スレ画面・カタログ共通の構造問題)

[P2-1] エッジスワイプ更新: オーバースクロール値をコンポジションで読んでいる
  場所: shared/src/commonMain/kotlin/ui/board/EdgeSwipeRefreshSupport.kt:122-148
        (特に :142 の remember(overscrollOffset))
  影響箇所: CatalogItemComponents.kt:39,50(カタログ Grid/List)
            ThreadContentLayout.kt:71-97(スレ画面)
  内容:
    - animateFloatAsState の値をコンポジション内で読み、バインディングを
      毎フレーム作り直している。上下端で引っ張った瞬間とリリース後の
      スプリング収束まで、毎フレーム:
        * CatalogGrid / ThreadContent 全体の再コンポーズ
        * offset { } / edgeSwipeRefresh(composed)のモディファイアチェーン再構築
        * リストの再レイアウト
    - Modifier.offset { } のラムダ版を使っているのに読み取りがコンポジション側に
      あるため、遅延読み取りの利点が失われている。
    - さらに onOverscrollTargetChanged はドラッグ中の毎スクロールフレームで
      呼ばれる(:314, :335)。
    - スレ画面はカードが重いため被害がカタログより大きい。
  修正方針:
    - overscrollOffset を State<Float>(または () -> Float)のまま offset { }
      ラムダ内で読む。バインディングを毎フレーム再生成しない。

[P2-2] スクロール端の判定もコンポジションで読んでいる
  場所: shared/src/commonMain/kotlin/ui/board/EdgeSwipeRefreshSupport.kt:91-103, 106-119
  内容:
    - derivedStateOf で作った isAtTop/isAtBottom をすぐコンポジションで読んで
      Boolean に潰しているため、最上部からスクロール開始した瞬間・最下部到達の
      瞬間に CatalogGrid / ThreadContent が再コンポーズされる。
      「スクロールし始めに一瞬引っかかる」症状の典型原因。
  修正方針:
    - edgeSwipeRefresh 側は rememberUpdatedState で受けているので、
      () -> Boolean のラムダで渡せばコンポジション読み取り自体が不要。

[P2-3] スレ画面: isScrollInProgress のコンポジション読み取り
  場所: shared/src/commonMain/kotlin/ui/board/ThreadContentLayout.kt:76
  内容:
    - val isScrolling = listState.isScrollInProgress をコンポジションで直接
      読んでいるため、スクロールの開始と停止のたびに ThreadContent 全体が
      再コンポーズされる。
    - その際 showQuotePreview が新インスタンスになり、items 内の
      buildThreadScreenPostCardCallbacks(:155-168)が全可視レス分作り直され、
      全 ThreadPostCard が再コンポーズされる。レスカードは画像+注釈付き
      テキストで重く、スクロール開始の引っかかりの主因。
  修正方針:
    - isScrolling は引用プレビュー抑制にしか使っていないので、ラムダ内で
      listState.isScrollInProgress を直接読む形にする。

[P2-4] スレ画面スクロールバー: layoutInfo をコンポジションで毎フレーム読む
  場所: shared/src/commonMain/kotlin/ui/board/ThreadContentSupport.kt:164-217
  内容:
    - listState.layoutInfo / firstVisibleItemScrollOffset をコンポジションで
      直接読んでいるため、ThreadScrollbar はスクロール中毎フレーム再コンポーズ
      される。毎フレーム visibleItems の map/filter/average のアロケーションも発生。
  修正方針:
    - 計算を Canvas の draw スコープ内(または drawWithCache + derivedStateOf)
      に移し、コンポジション読み取りをなくす。

[P3-4] サムネ失敗時の動画フォールバックが重い
  場所: shared/src/commonMain/kotlin/ui/image/ImageLoaderProvider.kt:96-147
        (FutabaExtensionFallbackInterceptor)
        shared/src/androidMain/kotlin/ui/image/ImageLoaderProvider.android.kt:10

[P3-7] カタログ: フィルタ結果が非同期で初期値が空リスト(ちらつき)

[B-1] Wear: スナップショット受信のたびに画面選択がリセットされる(重要度: 高)
  場所: app-wear/src/main/java/com/valoser/futacha/wear/WearMainActivity.kt:97
  内容:
    - var selectedThreadId by remember(snapshot?.generatedAtMillis) { mutableStateOf(null) }
      のキーが generatedAtMillis のため、新しいスナップショットが届くたびに
      選択状態が null にリセットされる。
    - スマホ側は読み上げ位置の変化のたびにスナップショットを再送する
      (ThreadScreen.kt:629 の LaunchedEffect → WatchReadAloudStatusStore.update
      → WatchSyncManager の combine が 1 秒デバウンスで送信)。
    - 結果: 読み上げ中はレスが進むたびに Watch の ReadAloudControlView が閉じて
      HomeView に強制的に戻される。ユーザーに見える不具合。
  修正方針:
    - キーを外す(remember { ... })。スレ消滅時は既に
      firstOrNull { it.threadId == selectedThreadId } で安全に処理されている。

[B-2] Cookie トランザクションがネットワーク I/O 中ロックを保持(重要度: 中)
  場所: shared/src/commonMain/kotlin/network/PersistentCookieStorage.kt:247-267
  内容:
    - commitOnSuccess は transactionMutex.withLock { block() } で block()
      (ネットワーク要求を含む)の完了までロックを保持する。
    - addCookie / removeCookie / clearAll は withMutationSerialization 経由で
      同じ transactionMutex を取るため、トランザクション外の並行リクエストが
      Set-Cookie を受け取るとトランザクションの通信終了までブロックされる。
      低速回線では Cookie 書き込みが直列化し、フリーズに見える遅延になり得る。
    - さらに block() 内で CookieTransactionContext を継承しない別の
      CoroutineScope に処理を launch して await し、その中で addCookie が
      呼ばれるとデッドロックする。
  修正方針:
    - ステージング用マップを分け、ロック保持を mutex 内のメモリ操作だけに限定する。

[B-3] ThreadScreen: AI コマンド実行 Effect のキー設計による二重実行(重要度: 中)
  場所: shared/src/commonMain/kotlin/ui/board/ThreadScreen.kt:816-917
  内容:
    - LaunchedEffect(args.aiCommand, readAloudSegments.size) で、消費通知
      (onAiCommandConsumed)が when 分岐の完了後にしか呼ばれない。
    - サスペンドする分岐(例: drawerState.open() はアニメーション完了まで
      suspend)の実行中に readAloudSegments.size が変化すると、Effect が
      未消費の同じコマンドでリスタートして再実行される。
    - 同期分岐は中断点がないため安全。suspend を含む分岐のみリスク。
  修正方針:
    - 消費判定を Effect 冒頭で行う(実行前に消費済みフラグを立てる)か、
      セグメント待機が必要なコマンドだけ別 Effect に分ける。
    - 同様の Effect はカタログ側にもある(CatalogScreen.kt:412-474)が、
      こちらはキーが args.aiCommand のみでリスクは低い。

[B-4] WatchSyncManager: 同期コレクタが非キャンセル例外で静かに死ぬ(重要度: 中)
  場所: app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt:124-128
  内容:
    - catch (error: Throwable) { if (error is CancellationException) throw error }
      が CancellationException 以外を握りつぶし、finally で isStarted=false に
      なるが誰も再起動しない。
    - combine の上流(ストレージ読み取り等)が例外を投げると、ログも出ずに
      Watch への自動スナップショット送信がプロセス再起動まで完全に停止する。
  修正方針:
    - FutachaApplication の背景フロー(retryWhen + バックオフ + ログ)と同様の
      リトライを入れる。

[B-5] ImagePicker(iOS): 30 秒タイムアウトの dispatch_after がキャンセルされない(低)
  場所: shared/src/iosMain/kotlin/util/ImagePicker.ios.kt:88-102
  内容:
    - 読み込みが即成功してもタイマーは 30 秒後に必ず発火し complete(null) を呼ぶ。
      ResumeGate で二重 resume は防がれているためクラッシュはしないが、
      クロージャ(と選択メディアへの参照)が 30 秒間保持され、毎回警告ログが出る。
  修正方針: DispatchWorkItem にしてキャンセルする。

[B-6] HistoryRefresher: バッチ失敗時に残りジョブがキャンセルされない(低)
  場所: shared/src/commonMain/kotlin/service/HistoryRefresher.kt:235-262
  内容:

  修正方針: 例外時に batchJobs.forEach { it.cancel() }。

[B-7] iOSApp.swift: BGTask 登録がリフレクション頼み(低)
  場所: iosApp/iosApp/iOSApp.swift:543-564
  内容:
    - NSClassFromString によるクラス名探索のため、共有フレームワークのクラス名が

  修正方針: shared を import 済みなので

[B-8] Wear: 同期ラベル・読み上げ鮮度判定が System.currentTimeMillis() 固定(低)
  場所: app-wear/src/main/java/com/valoser/futacha/wear/WearMainActivity.kt:473-495

■ 改善済み

[DONE-1] [P1-1] iOS WebM サムネイル生成に全体タイムアウトを追加
  対応日: 2026-06-11
  場所: shared/src/iosMain/kotlin/ui/image/IosVideoFrameFetcher.kt:216-255
  改善したこと:
    - WebM サムネイル生成の WKWebView 待機全体を 10 秒で打ち切るようにした。
    - didFinishNavigation / didFailNavigation が呼ばれない停止状態でも、
      suspendCancellableCoroutine がキャンセルされ、invokeOnCancellation 経由で
      WebmThumbnailNavigationDelegate.dispose() が main queue 上で実行される。
