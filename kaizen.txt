Futacha performance kaizen notes
Date: 2026-06-23

Scope:
- This file is a static-code-review backlog for catalog/thread performance work.
- It is not referenced by the app at runtime.
- Obsolete findings and already-completed items were removed on 2026-06-23.
- No real-device profiling has been performed yet, so priorities are based on code structure.
- Do not add new runtime logging, debug timing logs, or diagnostic file/console output for any kaizen item. Logging itself consumes I/O and can become the performance problem being investigated.
- Prefer deterministic tests, code-path reduction, existing persisted state, bounded in-memory counters used only by tests, and manual profiling tools over adding app runtime logs.

Current summary:
- The largest remaining risks are response buffering, duplicate post identity handling, storage/cookie consistency, and unbounded external inputs.
- Several older findings are no longer valid: SavedThreadsScreen is wired, Catalog has Grid/List display modes, Thread LazyList keys are stable, embeddedHtml extraction is skipped on Android/iOS parser actuals, several small thumbnail requests now specify size() with crossfade disabled, history archive/export work is complete, catalog head metadata/title completion is policy-controlled, item 3 thread-wide derived work has been reduced enough to treat as complete, item 4 WatchWords mode now stages primary catalog display before merging additional sources, item 30 history persistence now coalesces to one latest revision write while skipping already persisted queued revisions, item 31 iOS storage writes now use separate locks for large JSON and scalar settings, item 32 AI availability now refreshes continuously only while a Thread screen is visible, item 33 read-aloud watch progress now uses dedicated small status updates instead of full snapshots, and item 34 iOS video preview now skips unchanged WKWebView JavaScript sync.
- Item 29 is intentionally skipped for now because replacing full-byte attachment posting with file/URI streaming is a broad API and platform-storage change.

Medium priority

26. Response reads can create avoidable memory and GC spikes
-----------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/network/HttpBoardApiResponseSupport.kt
- shared/src/commonMain/kotlin/parser/CatalogHtmlParserCore.kt
- shared/src/commonMain/kotlin/parser/ThreadHtmlParserCore.kt

Current behavior:
- Responses are accumulated into a growing ByteArray by copyOf expansion.
- The final ByteArray is decoded into String.
- Parsers may then normalize or slice the String, creating additional transient allocations.

Risk:
- A 5-10MB catalog/thread response can briefly exist as multiple large buffers and strings.
- On memory-constrained devices this can cause GC pauses, frame drops, or OOM-like failures during refresh.
- Head reads use a similar accumulation path, so repeated OP image helper requests can add smaller but frequent allocation pressure.

Kaizen:
- Do not add response-size runtime logs. Prefer parser/API tests and code changes that avoid duplicate buffering.
- Avoid extra full-string normalization copies where possible.
- Consider a streaming or segmented parse path for catalog/thread HTML before raising size limits.


35. Cookie snapshot writes are not serialized after leaving the storage mutex
----------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/network/PersistentCookieStorage.kt
- shared/src/commonMain/kotlin/network/PersistentCookieSnapshotSupport.kt
- shared/src/commonMain/kotlin/network/PersistentCookieTransactionSupport.kt

Current behavior:
- Cookie map mutation is protected by PersistentCookieStorage.mutex.
- The JSON snapshot is created while locked, but persistSnapshot() runs after the mutex is released.
- persistPersistentCookieSnapshot() reads the current primary file, writes a backup, then writes the new primary.
- withMutationSerialization currently just executes block() and does not serialize disk writes.

Risk:
- Two concurrent cookie mutations can create snapshots A and B, then write them to disk out of order.
- The in-memory cookie map is correct, but a later process/app restart can restore an older snapshot if the older disk write finishes last.
- Backup creation can also capture an intermediate primary, which makes post-cookie recovery harder to reason about.

Kaizen:
- Add a dedicated persist mutex around cookie snapshot disk writes.
- Include a monotonically increasing cookie snapshot revision in the file and ignore older writes if possible.
- Add stress tests for concurrent add/remove/transaction commits followed by reload.


36. Cookie initialization failures can add setup work to every board request
---------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/repo/BoardRepository.kt
- shared/src/commonMain/kotlin/repo/DefaultBoardRepositoryExecutionSupport.kt
- shared/src/commonMain/kotlin/repo/DefaultBoardRepositorySupport.kt
- shared/src/commonMain/kotlin/network/HttpBoardApi.kt

Current behavior:
- ensureCookiesInitialized() checks initializedBoards and existing posting cookies.
- If no valid cookie exists, it calls fetchCatalogSetup(board).
- When setup fails or does not produce a preferred cookie, the board is not marked initialized.
- The next catalog/thread/helper request repeats the setup attempt.

Risk:
- A board that rejects setup, has blocked cookies, or is temporarily offline can pay the catalog-setup request before every primary request.
- Catalog display and OP image/title helper fetches can all queue behind the same per-board initialization lock.
- Users may see repeated slow loads instead of one clear "cookie setup unavailable" state.

Kaizen:
- Add a short negative cache for failed cookie initialization per board.
- Do not add cookie setup runtime logs. Track recent setup failures in memory if needed for behavior, and test lock/setup decisions.
- Let read-only catalog/thread loads proceed after a recent setup failure while still surfacing the cookie warning.


37. OP image/title helper timeout covers both semaphore wait and network fetch
------------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/repo/BoardRepository.kt
- shared/src/commonMain/kotlin/repo/DefaultBoardRepositoryExecutionSupport.kt
- shared/src/commonMain/kotlin/ui/board/CatalogHeadMetadataSupport.kt

Current behavior:
- fetchDefaultBoardRepositoryOpImageWithPermit() wraps semaphore.withPermit { fetch() } in withTimeoutOrNull(30s).
- resolveCatalogDisplayTitle() uses the same opImageSemaphore and timeout.
- When the timeout fires, fetchOpImageUrl() returns null without saving a miss-cache entry.

Risk:
- A slow helper network fetch consumes one of the 4 permits until timeout.
- Timed-out helper requests are not negatively cached, so visible catalog movement can retry the same thread soon after.
- Because image and title helpers share the semaphore, slow title resolution can delay OP image completion and vice versa.

Kaizen:
- Split "wait for permit" timeout from "network helper request" timeout.
- Save a short miss/timeout cache entry for helper timeouts.
- Track separate semaphores or priority for visible image helpers versus title-only helpers.


38. Android SAF writes have no overall write/flush timeout
---------------------------------------------------------
Relevant code:
- shared/src/androidMain/kotlin/util/FileSystem.android.kt
- shared/src/commonMain/kotlin/com/valoser/futacha/shared/service/ThreadSaveService.kt
- shared/src/commonMain/kotlin/com/valoser/futacha/shared/service/ThreadSaveNetworkSupport.kt

Current behavior:
- TreeUri writes open a ContentResolver output stream and write ByteArray chunks of 64KB.
- writeByteStream() validates total size but relies on runInterruptible output.write()/flush()/close().
- There is no overall timeout around openOutputStream, write loop, flush, or close for SAF providers.

Risk:
- A slow or buggy third-party file manager/document provider can block a save operation for a long time.
- Cancellation is checked between chunks, but a blocking provider call may not return promptly.
- Manual save, autosave, and index writes can appear frozen while waiting on the provider.

Kaizen:
- Add an overall SAF write timeout, but do not log provider/package, byte count, or write phases.
- Prefer app-scoped temp file plus provider copy only for the final export path.
- Show a distinct "保存先プロバイダの応答待ち" message when SAF writes exceed a threshold.


39. iOS background refresh uses a separate HistoryRefresher instance from the UI
--------------------------------------------------------------------------------
Relevant code:
- shared/src/iosMain/kotlin/MainViewController.kt
- shared/src/iosMain/kotlin/background/BackgroundRefreshManager.kt
- shared/src/commonMain/kotlin/service/HistoryRefresher.kt
- shared/src/commonMain/kotlin/state/AppStateStore.kt

Current behavior:
- runIosBackgroundRefresh() constructs a new DefaultBoardRepository and a new HistoryRefresher for each BGTask run.
- HistoryRefresher prevents duplicate refresh only with an instance-local mutex.
- The iOS UI side can also refresh history through its own refresher while the BGTask refresher is active.
- Both paths share AppStateStore, cookie storage, HttpClient ownership, and auto-save repository.

Risk:
- Foreground UI refresh and iOS BGTask refresh can run at the same time, unlike Android where the app-scoped HistoryRefresher is shared.
- This can duplicate network requests, compete for cookie persistence, and interleave history/autosave writes.
- The final UI state may be correct due to AppStateStore locking, but users can still see extra load, battery use, and delayed saves.

Kaizen:
- Use a process-wide refresh gate on iOS, or inject the same HistoryRefresher into BGTask and UI where lifecycle permits.
- Add refresh source labels and active-refresh metrics.
- When the app is foreground, skip or downgrade BGTask refresh work.


40. iOS BGTask cancellation still performs non-cancellable repository close
--------------------------------------------------------------------------
Relevant code:
- shared/src/iosMain/kotlin/MainViewController.kt
- shared/src/iosMain/kotlin/background/BackgroundRefreshManager.kt
- shared/src/commonMain/kotlin/repo/BoardRepository.kt

Current behavior:
- BackgroundRefreshManager cancels the active job when iOS calls expirationHandler.
- runIosBackgroundRefresh() catches timeout/cancellation, then always enters finally.
- finally closes the temporary repository with withContext(NonCancellable) { repo.closeAsync().join() }.

Risk:
- After iOS has already expired the BGTask, cleanup can still spend time closing repository state.
- If close waits behind helper work or IO cleanup, the process may continue doing work past the useful background window.
- This can make expiration behavior harder to diagnose because logs show cancellation followed by more cleanup activity.

Kaizen:
- Bound repository close with a short timeout in BGTask cleanup.
- Avoid starting helper work that outlives the BGTask deadline.
- Do not add close-duration logs. Separate close from refresh control flow and cover cancellation behavior with tests.


41. Android DataStore read fallback can temporarily expose empty/default state
------------------------------------------------------------------------------
Relevant code:
- shared/src/androidMain/kotlin/state/AppStateStore.android.kt
- shared/src/commonMain/kotlin/state/AppStateStore.kt
- shared/src/commonMain/kotlin/ui/FutachaAppObservedRuntimeStateSupport.kt
- shared/src/commonMain/kotlin/ui/FutachaApp.kt

Current behavior:
- safeData caches the last readable Preferences snapshot.
- On IOException, it emits the cached snapshot while retrying.
- If no cached snapshot exists, catch emits emptyPreferences().
- FutachaApp observes these flows and also calls seedIfEmpty() at startup.

Risk:
- On a cold start with DataStore read failures, the UI can temporarily see empty/default values for boards, history, app lock, ads, background refresh, and settings.
- seedIfEmpty() only fills missing keys during edit, so it avoids normal overwrites, but the user-visible state can still flicker to defaults.
- App lock is especially sensitive: a transient null appLockPasswordHash can allow the main UI to appear before the real hash is readable.

Kaizen:
- Treat "no cached snapshot + DataStore read failure" as a loading/error state instead of emitting emptyPreferences().
- Keep app lock in a conservative locked/loading state until the first successful appLockPasswordHash read.
- Add a test for cold-start DataStore IOException before any readable snapshot exists.


42. AI command bridge can enqueue more work than the UI can apply coherently
---------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ai/FutachaAiCommandBridge.kt
- shared/src/commonMain/kotlin/ui/FutachaApp.kt
- shared/src/commonMain/kotlin/ui/FutachaAiCommandRouter.kt
- app-android/src/main/java/com/valoser/futacha/FutachaAppFunctionService.kt
- app-android/src/main/java/com/valoser/futacha/WatchSyncManager.kt

Current behavior:
- FutachaAiCommandBridge is a Channel with capacity 128.
- FutachaApp is intended to be the single collector.
- RefreshHistory commands from the bridge are launched in child coroutines, while most commands are handled sequentially.
- Deduplication only works when commandId is provided and remembered in an in-memory LinkedHashSet.

Risk:
- AppFunction/Watch callers can enqueue many distinct commands while the app is launching or locked.
- Multiple RefreshHistory commands can run concurrently until HistoryRefresher rejects/skips duplicates, still doing command routing and result handling work.
- Screen-forwarded commands share one pendingAiScreenCommand slot, so fast command bursts can overwrite earlier foreground actions before a screen consumes them.

Kaizen:
- Add per-action coalescing for RefreshHistory, screen-forwarded commands, and destructive confirmation commands.
- Replace the single pendingAiScreenCommand slot with a small typed queue or reject a new foreground command while one is pending.
- Do not report queue depth through runtime logs. Expose queue depth/drop counts only to tests or user-visible state if product behavior needs it.


43. Thread load fallback waits remote, archive, and offline paths sequentially
-----------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/ThreadLoadRunnerSupport.kt
- shared/src/commonMain/kotlin/ui/board/ThreadScreenRunnerBindingsSupport.kt
- shared/src/commonMain/kotlin/ui/board/ThreadScreenRuntimeSupport.kt

Current behavior:
- Remote thread load waits up to THREAD_REMOTE_LOAD_TIMEOUT_MS = 20s.
- If fallback rules allow it, archive fallback waits up to 8s.
- Offline fallback then waits up to 5s.
- These waits are sequential, so a saved offline copy may not appear until all earlier paths finish or time out.

Risk:
- In poor networks, opening a thread with a valid local autosave can still look frozen for up to roughly 33s.
- The behavior overlaps with "prefer local stale display", but this is a separate timeout-chain problem: offline fallback is last even when local data exists.
- Manual refresh can also clear isShowingOfflineCopy before entering the same slow chain.

Kaizen:
- Probe local metadata quickly before remote load and display stale copy while remote/archive continue.
- Short-circuit archive fallback when a recent autosave exists and the primary failure is timeout/offline.
- Do not add fallback timing logs. Make fallback order explicit in state/results and cover it with tests.


44. Android AdView is created and loaded before it is visible
-----------------------------------------------------------
Relevant code:
- shared/src/androidMain/kotlin/ui/board/ThreadScreenBannerAd.android.kt
- shared/src/commonMain/kotlin/ui/board/ThreadScreenScaffoldHost.kt
- shared/src/commonMain/kotlin/ui/FutachaApp.kt

Current behavior:
- ThreadScreenBannerAd creates an AdView with remember().
- It delays 1.2s, calls loadAd(), and only inserts AndroidView after onAdLoaded sets isBannerVisible=true.

45. Android AppFunction service reports success after enqueue, not after execution
--------------------------------------------------------------------------------
Relevant code:
- app-android/src/main/java/com/valoser/futacha/FutachaAppFunctionService.kt
- shared/src/commonMain/kotlin/ai/FutachaAiCommandBridge.kt
- shared/src/commonMain/kotlin/ui/FutachaAiCommandRouter.kt
- shared/src/commonMain/kotlin/ui/FutachaApp.kt

Current behavior:
- FutachaAppFunctionService validates an AppFunction request and enqueues a FutachaAiCommand.
- It starts MainActivity and immediately returns an ExecuteAppFunctionResponse with queued status.
- Actual execution can later fail due to settings disabled, missing screen context, confirmation required, storage failure, or command queue overwrites.

Risk:
- External callers see "queued" even when the app cannot complete the action.
- Repeated callers may send more commands because they believe the previous one succeeded, increasing queue pressure.
- For destructive/confirmation commands, the mismatch between external success and in-app pending confirmation can be confusing.

Kaizen:
- Include a commandId in the response and expose a lightweight status/ack path.
- For commands that require confirmation or foreground screen state, return a more explicit "accepted_pending_user_action" status.
- Coalesce duplicate AppFunction commands by commandId/source before starting MainActivity again.


46. Thread tree mode can drop posts when post IDs are duplicated
---------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/ThreadTreeContent.kt
- shared/src/commonMain/kotlin/ui/board/ThreadPostSupport.kt

Current behavior:
- buildThreadTreeNodes() builds orderIndex, postIndex, parentByPostId, and visited by post.id.
- postIndex uses associateBy { it.id }, so duplicate IDs keep only one Post.
- visited is also keyed by post.id, so a second Post with the same ID is skipped in tree traversal.
- Flat thread rendering already has duplicate-aware lazy keys, but tree construction does not use that stable duplicate ordinal.

Risk:
- Malformed, recovered, or parser-edge-case threads with duplicate post IDs can render fewer posts in tree mode than in flat mode.
- Parent selection can point at the wrong duplicate because orderIndex and postIndex disagree with the original list.
- Missing posts in tree mode can also hide quotes, media, and moderation actions, making the issue look like data loss.

Kaizen:
- Build tree nodes around a stable post key that includes duplicate ordinal or original list index.
- Keep quote target resolution by canonical post ID, but map target IDs to all matching candidate nodes and choose the nearest earlier candidate.
- Add a unit test with duplicated post.id values and verify tree mode preserves all posts.


47. Quote preview composes every target post at once
--------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/ThreadQuotePreviewComponents.kt
- shared/src/commonMain/kotlin/ui/board/ThreadPostSupport.kt
- shared/src/commonMain/kotlin/parser/ThreadHtmlReferenceSupport.kt

Current behavior:

- ID references can resolve to every post from the same poster ID, and media/text references can also produce multi-post target sets.

48. Thread filter cache stores whole ThreadPage copies
-----------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/ThreadScreenContentHost.kt
- shared/src/commonMain/kotlin/ui/board/ThreadScreenMutableStateSupport.kt
- shared/src/commonMain/kotlin/ui/board/ThreadFilterSupport.kt
- shared/src/commonMain/kotlin/ui/board/ThreadScreenRuntimeSupport.kt

Current behavior:
- threadFilterCache is a per-thread LinkedHashMap with THREAD_FILTER_CACHE_MAX_ENTRIES = 8.
- Each entry stores a full ThreadPage copy containing a filtered posts list.
- Cache keys include post fingerprint, NG state, filter options, keyword, self identifiers, and sort state.
- When a filter is active and no cache entry exists, the initial filteredPage is state.page.copy(posts = emptyList()) until background parsing finishes.

Risk:
- Rapid keyword edits or NG/filter toggles can keep several full filtered post lists alive for the same large thread.
- Repeated ThreadPage copies increase allocation and GC pressure in the screen that is already doing search, highlight, image, and scroll work.
- The empty-list initial value can briefly blank the thread during recomputation, which looks like a rendering glitch or short freeze.

Kaizen:
- Cache only filtered post IDs, original indices, or small filter results instead of whole ThreadPage objects.
- Keep the previous filtered result visible while recomputing a new key, with an explicit loading flag if needed.
- Clear or coalesce keyword-derived cache entries during active typing.


49. Cache cleanup actions can run concurrently with active image cache users
---------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/GlobalSettingsBindingsSupport.kt
- shared/src/commonMain/kotlin/ui/board/GlobalSettingsRuntimeSupport.kt
- shared/src/commonMain/kotlin/ui/image/ImageLoaderProvider.kt

Current behavior:
- Global settings exposes clearImageCache and clearTemporaryCache as independent buttons.
- Each action launches a coroutine, but there is no shared in-progress guard, mutex, or timeout.
- clearImageCache calls imageLoader.diskCache?.clear() and memoryCache?.clear().
- clearTemporaryCache deletes resolveImageCacheDirectory(platformContext), which is the same futacha_image_cache directory used to build the Coil DiskCache.

Risk:
- Repeated taps can start overlapping cache clear/delete operations.
- Deleting the disk cache directory while the ImageLoader or DiskCache is reading/writing it can cause IO errors, cache misses, or long stalls.
- A large cache directory delete has no operation timeout and can keep the settings coroutine busy for an unexpectedly long time.

Kaizen:
- Gate cache cleanup behind one screen-level cacheCleanupJob or Mutex and disable buttons while it runs.
- Prefer DiskCache.clear() for the active image cache; only delete directories after the ImageLoader/DiskCache is closed or recreated.
- Add a timeout for cache cleanup, but do not add progress/error runtime logging.


50. File manager picker does broad PackageManager and icon work on every dialog open
----------------------------------------------------------------------------------
Relevant code:
- shared/src/androidMain/kotlin/util/FileManagerPicker.android.kt
- shared/src/androidMain/kotlin/ui/board/FileManagerPickerDialog.android.kt

Current behavior:
- getAvailableFileManagers() queries PackageManager with five broad intents using MATCH_ALL.
- It loads labels and Drawable icons for every resolved activity.
- FileManagerItem then converts each Drawable to a 48x48 bitmap in per-row produceState.
- Results are remembered only for the current dialog composition.

Risk:
- On devices with many document providers or file managers, opening the settings picker can take noticeable time.
- Icon conversion starts per visible row and can keep doing work while the dialog is being scrolled or reopened.
- Reopening the dialog repeats the PackageManager query and icon conversion, creating avoidable settings-screen jank.

Kaizen:
- Cache the resolved file manager list at the settings-runtime level and invalidate it only when package changes are relevant.
- Load lightweight label/package rows first, then lazy-load icons with a small icon cache.
- Avoid MATCH_ALL where a narrower query is enough, or cap/de-duplicate earlier before loading labels/icons.


51. Cookie manager allows overlapping delete, clear, and reload operations
-------------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/CookieManagementScreen.kt
- shared/src/commonMain/kotlin/repository/CookieRepository.kt
- shared/src/commonMain/kotlin/network/PersistentCookieStorage.kt

Current behavior:
- Each row delete launches a new coroutine, calls repository.deleteCookie(), then reload(), then showSnackbar().
- The clear-all button launches the same style of independent coroutine.
- reload() has generation protection for applying results, but delete/clear buttons are not disabled while a mutation is in progress.
- PersistentCookieStorage serializes internal state with a Mutex, but snapshots are persisted after the lock is released.

Risk:
- Fast repeated taps can enqueue multiple cookie mutations and full reloads.
- UI reloads may be skipped by generation checks, but the underlying file writes and cookie-store updates still run.
- Snackbars can report deletes after the visible cookie list has already moved on, making the screen feel inconsistent.

Kaizen:
- Add a mutationInProgress state and disable row delete/clear-all actions while a cookie mutation is running.
- Serialize CookieManagementScreen mutations through one screen-level job or queue.
- For multiple deletes, offer a batch action that mutates once and reloads once.


52. Archive search response body is decoded without a size cap
--------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/network/ArchiveSearch.kt
- shared/src/commonMain/kotlin/ui/board/CatalogScreenExecutionSupport.kt

Current behavior:
- searchInqueuetArchiveThreads() calls httpClient.get(.../search).
- It then reads the whole response with bodyAsText().
- The result limit is capped to MAX_ARCHIVE_SEARCH_LIMIT = 100, but the raw HTTP response body itself is not capped before allocation and JSON decode.
- HttpBoardApi fetch paths have explicit body/header size checks, but this archive-search path does not.

Risk:
- A server bug, proxy error page, or maliciously large response can allocate a large String and then feed it to kotlinx.serialization.
- Past-thread search runs from the catalog UI, so this can surface as a search sheet freeze or process memory spike.
- Error responses are also read fully before truncating the displayed detail to 120 characters.

Kaizen:
- Read archive search responses through the same bounded channel helper style used by HttpBoardApi.
- Add a small archive-search JSON body cap, for example 1-2MB, because only up to 100 results are expected.
- Truncate or stream error bodies before creating the exception message.


53. Archive thumbnail enrichment starts one coroutine per search result
----------------------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/network/ArchiveSearch.kt
- shared/src/commonMain/kotlin/ui/board/CatalogSearchDialogComponents.kt

Current behavior:
- enrichArchiveSearchItemsWithThumbnails() maps every ArchiveSearchItem to async { ... } and awaitAll().
- Each item missing thumbUrl can fetch the archive thread page head.
- Each thumbnail probe can read up to 96KB and wait up to 10s.
- The search result limit can be as high as 100.

Risk:
- A past-thread search with many items missing thumbnails can start up to 100 concurrent head requests.
- Even with per-request limits, this can cause network bursts, socket pressure, and slow cancellation when the user changes query or dismisses the sheet.
- The UI waits for all thumbnail probes before publishing the Success state, so visible text results are delayed by slow thumbnail lookups.

Kaizen:
- Limit thumbnail enrichment concurrency with a Semaphore, for example 4.
- Publish archive search text results immediately and fill thumbnails lazily in the result list.
- Skip thumbnail enrichment on metered/slow networks or after a small total enrichment budget.


54. Thread AI summary cache ignores refreshed post content
---------------------------------------------------------
Relevant code:
- shared/src/commonMain/kotlin/ui/board/ThreadScreenContentHost.kt
- shared/src/commonMain/kotlin/ui/board/ThreadAiCacheSupport.kt

Current behavior:
- aiCacheKey includes postsFingerprint, but summaryState is keyed by threadSummaryCacheKey.
